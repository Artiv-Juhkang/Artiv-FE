# 알림 센터 (Notification) — 상세 설계 (데이터 모델까지)

> [`roadmap-and-monetization.md`](roadmap-and-monetization.md)의 **최우선 받침대**. 인앱 DB 알림 1개로 신규회차·문의답변·(향후)작가소식·커뮤니티 반응을 모두 처리.
> 원칙: **제네릭 이벤트 로그** — 새 알림원이 생겨도 스키마 무변경, enum+이벤트+리스너만 추가.
> 상태: 설계 확정용(미구현). 구현 시 이 문서를 스펙으로 사용. 외부 푸시(FCM/APNs)는 범위 밖.

---

## 1. 범위 & 목표
- **하는 것**: 인앱 알림 저장·조회·읽음, fan-out 생성(구독자/대상자), 미읽음 뱃지.
- **안 하는 것(지금)**: 외부 푸시, 실시간(SSE/WS)은 폴링으로 시작하되 확장 훅만 남김, 알림 설정(뮤트)은 2단계.
- **발화 가능(현재 코드)**: `EPISODE_PUBLISHED`(구독작 신규회차), `INQUIRY_ANSWERED`(문의 답변). 나머지는 해당 도메인이 생길 때 추가.

## 2. 알림 타입 분류 (제네릭)
| type (enum) | 수신자(recipient) | 유발자(actor) | target | 발화 지점 | 상태 |
|---|---|---|---|---|---|
| `EPISODE_PUBLISHED` | 작품 구독자 | 작가 | EPISODE | 즉시발행/예약발행 | **지금** |
| `INQUIRY_ANSWERED` | 문의 작성자 | 관리자 | INQUIRY | 답변 PATCH | **지금** |
| `AUTHOR_POST` | 작가 팔로워 | 작가 | POST | 작가 소식 작성 | 향후(팔로우) |
| `POST_COMMENT` / `COMMENT_REPLY` | 글/댓글 작성자 | 댓글 작성자 | COMMENT | 댓글·대댓글 | 향후(커뮤니티) |
| `POST_BEST` | 글 작성자 | 시스템 | POST | 베스트 승격 | 향후 |
| `FOLLOWED` | 작가 | 독자 | USER | 팔로우 | 향후 |

→ **새 타입 추가 = enum 값 + 도메인 이벤트 + 리스너 분기.** 테이블/마이그레이션은 그대로.

## 3. 데이터 모델 (핵심)

### 3.1 설계 결정 (왜 이렇게)
- **폴리모픽 타겟** (`target_type` + `target_id`): 타입마다 컬럼을 늘리지 않음 → 확장성. FK는 안 검(이벤트 로그라 관계무결성보다 유연성·삭제 견딤 우선).
- **렌더 필드 비정규화** (`title`, `message`): 알림이 **조인 없이 단독 렌더**, **대상이 삭제돼도 표시**됨(시점 스냅샷). 트레이드오프: 원본 제목이 바뀌어도 과거 알림은 옛 제목 유지(이벤트 로그라 허용).
- **recipient_id를 연관(User) 대신 `Long`**: 대량 fan-out(구독자 수천) 시 엔티티 그래프 로딩 회피 → 배치 insert가 싸다. 알림은 recipient_id로만 조회하므로 연관 불필요. **의도적 비연관**(엔티티 규칙의 정당한 예외).
- **`read_at timestamptz` (null=미읽음)**: 단일 컬럼이 읽음여부+읽은시각 동시 표현(`read`는 예약어 회피).
- **`dedup_key` + 부분 유니크**: 멱등(예약발행 스케줄러 재실행 시 구독자당 회차 1건 보장). 중복 허용 이벤트(댓글 등)는 null.

### 3.2 마이그레이션 — `V15__create_notifications.sql`
> (V14는 장르·태그가 사용 → 알림은 V15. 구현 시점의 다음 가용 버전을 부여.)
```sql
-- 알림(인앱). 제네릭 이벤트 로그: 폴리모픽 타겟 + 비정규화 렌더필드.
create table notifications (
    id           bigint        generated by default as identity primary key,
    recipient_id bigint        not null references users (id),  -- 받는 사람
    type         varchar(40)   not null,                        -- EPISODE_PUBLISHED ...
    target_type  varchar(20),                                   -- SERIES/EPISODE/INQUIRY/POST/COMMENT/USER
    target_id    bigint,                                        -- 클라 라우팅용 식별자(FK 안 검)
    actor_id     bigint,                                        -- 유발자(작가/관리자). null=시스템
    title        varchar(255)  not null,                        -- 비정규화 헤드라인
    message      varchar(500)  not null,                        -- 비정규화 본문
    read_at      timestamptz,                                   -- null=미읽음
    dedup_key    varchar(200),                                  -- 멱등키. null=중복 허용
    created_at   timestamptz   not null,
    updated_at   timestamptz   not null
);

-- 내 목록(최신순 페이징)
create index idx_notifications_recipient on notifications (recipient_id, created_at desc);
-- 미읽음 개수(부분 인덱스)
create index idx_notifications_unread on notifications (recipient_id) where read_at is null;
-- 멱등 — 수신자 단위(있을 때만 유니크). 전역이면 다대상 fan-out에서 교차 수신자 묵음(STEP 24 리뷰 반영).
create unique index uq_notifications_dedup on notifications (recipient_id, dedup_key) where dedup_key is not null;
```

### 3.3 엔티티
```java
@Entity @Table(name = "notifications")
@Getter @NoArgsConstructor(access = PROTECTED)
public class Notification extends BaseEntity {          // created_at/updated_at
    @Id @GeneratedValue(strategy = IDENTITY) Long id;

    @Column(name = "recipient_id", nullable = false) Long recipientId;   // 비연관(대량 fan-out)
    @Enumerated(STRING) @Column(nullable = false, length = 40) NotificationType type;
    @Enumerated(STRING) @Column(name = "target_type", length = 20) NotificationTargetType targetType;
    @Column(name = "target_id") Long targetId;
    @Column(name = "actor_id")  Long actorId;
    @Column(nullable = false, length = 255) String title;
    @Column(nullable = false, length = 500) String message;
    @Column(name = "read_at")   Instant readAt;
    @Column(name = "dedup_key", length = 200) String dedupKey;

    // 정적 팩토리: create(recipientId, type, targetType, targetId, actorId, title, message, dedupKey)
    public void markRead() { if (readAt == null) readAt = Instant.now(); }
    public boolean isRecipient(Long userId) { return recipientId.equals(userId); }
}
```
```java
enum NotificationType { EPISODE_PUBLISHED, INQUIRY_ANSWERED /*…향후*/ }
enum NotificationTargetType { SERIES, EPISODE, INQUIRY, POST, COMMENT, USER }
```

## 4. fan-out & 이벤트 (도메인 이벤트 기반)

> 이벤트 패턴은 **이 프로젝트 최초 도입**. 발행/답변 도메인이 알림을 **직접 모르게** 디커플링.

### 4.1 도메인 이벤트
```java
record EpisodePublishedEvent(Long seriesId, Long episodeId, int episodeNo, String seriesTitle, Long authorId) {}
record InquiryAnsweredEvent(Long inquiryId, Long recipientId, String inquiryTitle) {}
```

### 4.2 발행 지점 (기존 코드에 publishEvent만 추가)
- `EpisodeService.upload` — **즉시발행(PUBLISHED)** 일 때만 `EpisodePublishedEvent` 발행. (SCHEDULED는 발행 안 됨 → 아래 예약에서)
- `EpisodeService.publishDueEpisodes` — `SCHEDULED→PUBLISHED` 전환되는 각 회차마다 발행. (스케줄러 `EpisodePublisher.publishDue`가 호출)
- `InquiryService.answer` — `InquiryAnsweredEvent(recipientId = inquiry.user.id)` 발행.

### 4.3 리스너 (AFTER_COMMIT + 비동기)
```java
@Component @RequiredArgsConstructor
class NotificationFanoutListener {
    @Async                                              // 발행 응답을 막지 않음
    @TransactionalEventListener(phase = AFTER_COMMIT)   // 커밋된 발행에만 알림(롤백 시 미발송)
    public void onEpisodePublished(EpisodePublishedEvent e) {
        List<Long> subscriberIds = subscriptionRepository.findUserIdsBySeriesId(e.seriesId());
        notificationService.fanOut(subscriberIds, NotificationType.EPISODE_PUBLISHED,
            NotificationTargetType.EPISODE, e.episodeId(), e.authorId(),
            e.seriesTitle(), e.episodeNo() + "화가 올라왔어요",
            uid -> "EP_PUB:" + e.episodeId() + ":" + uid);   // dedupKey
    }
    @Async @TransactionalEventListener(phase = AFTER_COMMIT)
    public void onInquiryAnswered(InquiryAnsweredEvent e) {
        notificationService.fanOut(List.of(e.recipientId()), NotificationType.INQUIRY_ANSWERED,
            NotificationTargetType.INQUIRY, e.inquiryId(), null,
            "문의 답변", "문의 ‘" + e.inquiryTitle() + "’에 답변이 등록됐어요", null);
    }
}
```
- **isolation**: AFTER_COMMIT이라 알림 실패가 발행 트랜잭션을 되돌리지 않음(원하는 격리). 실패는 로깅/재시도 대상이지 발행을 막지 않음.
- **새 트랜잭션**: `@Async`로 별 스레드 → `fanOut`은 `@Transactional(REQUIRES_NEW)`로 배치 insert.
- **설정 필요**: `@EnableAsync` + `TaskExecutor` 빈.

### 4.4 멱등 & 성능 (배치)
- **멱등**: `dedup_key` 부분 유니크. 배치 insert는 **`ON CONFLICT (dedup_key) DO NOTHING`**(네이티브/`JdbcTemplate.batchUpdate`)로 재실행 시 중복 스킵.
- **성능**: 구독자 수천이면 `saveAll`(N insert) 대신 **JdbcTemplate 배치 insert** 권장. 소규모 시작은 `saveAll` 허용.
- **리포지토리 추가**: `SubscriptionRepository`에
  `@Query("select s.user.id from Subscription s where s.series.id = :seriesId") List<Long> findUserIdsBySeriesId(Long seriesId)`.

## 5. API & DTO
| 메서드·경로 | 용도 |
|---|---|
| `GET /api/me/notifications?page&size` | 내 알림 목록(최신순 페이징) → `PageResponse<NotificationResponse>` |
| `GET /api/me/notifications/unread-count` | 미읽음 개수(뱃지) → `{ count }` |
| `PATCH /api/me/notifications/{id}/read` | 하나 읽음(수신자 본인만) → 204 |
| `PATCH /api/me/notifications/read-all` | 전체 읽음(bulk update) → 204 |

```java
record NotificationResponse(Long id, NotificationType type, NotificationTargetType targetType,
                            Long targetId, Long actorId, String title, String message,
                            boolean read, Instant createdAt) {} // read = readAt != null
```
- 권한: 전부 `@AuthenticationPrincipal Long userId` 기준. `markRead`는 로드 후 `isRecipient(userId)` 검증(아니면 403). `read-all`은 `@Modifying update ... where recipient_id=:uid and read_at is null`.
- **클라 라우팅**: `(targetType, targetId)`로 딥링크 구성(예: EPISODE→회차 화면). 링크는 저장 안 함(라우트 변경에 강함).

## 6. 전달 방식 — 폴링부터
- **MVP**: 클라가 `unread-count`를 주기 폴링 + 알림함 열 때 목록 조회. 결제·인프라 0.
- **확장 훅(향후)**: 리스너가 fan-out 시 SSE Emitter 레지스트리에도 push하면 실시간으로 승격 가능(읽기 모델은 동일). WebSocket/외부푸시는 [`ToDoList.md`](../ToDoList.md) 표 참조.

## 7. 운영 / 엣지
- **보존정책**: 무한 증가 → `@Scheduled` 정리(읽음 + 90일 경과 삭제) 향후 추가.
- **대상 삭제**: 폴리모픽이라 FK 없음 → 대상(회차/문의) 삭제 시 알림은 비정규화 필드로 **여전히 렌더**, 클릭만 404 가능(클라가 "삭제된 콘텐츠" 처리). 또는 정리 배치.
- **알림 설정(뮤트)**: 2단계 — `notification_preferences(user, type, enabled)` + 리스너가 생성 전 확인. 지금은 전체 on.
- **자기 알림 제외**: 작가가 자기 작품 구독했어도 보통 제외(선택). 구독자 목록에서 `authorId` 제외 옵션.

## 8. 새 알림원 추가 절차 (확장성 증명)
1. `NotificationType`에 값 추가 → 2. 해당 도메인에서 이벤트 발행 → 3. 리스너에 분기 추가(타겟·문구·dedup). **스키마·API 무변경.**
